## 为什么

AIRI 已经提供人格、记忆和表现层，而 OpenClaw 专注于工具执行与自动化。此变更定义了一条清晰的集成路径，让 AIRI 能将可执行任务委托给 OpenClaw，同时不破坏角色一致性，也不向面向用户的层暴露执行细节。

## 变更内容

- 在 AIRI 与 OpenClaw 之间增加一层桥接，用于任务委托和结构化结果返回。
- 为任务路由、执行状态和最终回复组合定义请求/响应契约。
- 保持 AIRI 负责意图判断和具有人设风格的回复，而 OpenClaw 负责工具、自动化和多步骤执行。
- 将面向关系的记忆与面向执行的任务历史分离。

## 能力

### 新增能力
- `airi-openclaw-bridge`：AIRI 可以将类任务请求路由到 OpenClaw，接收结构化执行结果，并将其转换为符合人设一致性的回复。

### 修改的能力
- `AIRI 对话编排与工具路由`：现有 `stage-ui` 编排流程需要在保持 MCP 与 xsai 工具调用机制不变的前提下，新增一个面向后端服务适配层的 OpenClaw 委托触点，用于把任务型请求路由到桥接服务，并在结果返回后继续完成 AIRI 侧的人设化响应，而不是在前端对话层承担 OpenClaw 的传输、认证或执行生命周期。

## 影响

影响 `stage-ui` 中现有对话编排触点与 server-channel 路由方式、一个位于 `services/` 一侧并负责 OpenClaw 传输 / 认证 / 执行生命周期 / 执行历史存储的适配桥接层、结构化结果的人设化回复流程，以及与该集成路径对应的文档和 OpenSpec 工件。
